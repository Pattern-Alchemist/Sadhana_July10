## 1. Project Summary

AstroKalki is a Next.js 16 (App Router) + React 19 + TypeScript "museum-grade" digital archive of
Indic contemplative practices (siddhis), primary manuscripts, cosmology, lineage maps, and a
5-tradition comparative matrix — plus a large suite of client-side ritual/practice tools (japa
counter, breath trainer, chakra scanner, yantra builder, dream/journal logs, an AES-encrypted local
"vault", and a tarot-style Oracle module). Data is served from PostgreSQL via Drizzle ORM with a
self-healing seed/bootstrap layer, and a parallel offline `knowledge/` corpus (Markdown + SQLite +
JSONL chunks) backs a keyword/FTS "Archivist" search plus a stubbed pgvector semantic-search
endpoint. The target audience is spiritually curious, scholarship-minded practitioners; the product
deliberately catalogues rather than prescribes. Styling is Tailwind v4 + Framer Motion, with no
external analytics or telemetry — all personal data stays client-side, encrypted at rest.

## 3. Enhancement Blueprint

### 3.1 Architecture & Modularity

- **Issue**: `Arena_Export/` fully duplicated `app/`, `components/`, `lib/`, `docs/`, `knowledge/`
  (393 files, ~18MB uncompressed + 3 zip bundles ~28MB more) inside version control.
- **Impact**: Every future change has to be made twice (or drifts silently — the two `page.tsx`
  copies had already diverged). Clones are slower, `git blame`/search is noisier, CI checks out
  46MB it doesn't need.
- **Solution**: ✅ **Applied** — `git rm -r Arena_Export Arena_Export-1.zip Arena_Export-2.zip
  Arena_Export-combined.zip`. If an "Arena export" snapshot is needed again, generate it on demand
  via a script rather than committing it.
- **Priority**: P0 · **Effort**: S

- **Issue**: `db/schema.ts` and `db/schema-v2.ts` both exist with no documented migration path or
  deprecation note.
- **Impact**: New contributors (human or AI) can't tell which schema is live, risking writes against
  the wrong shape.
- **Solution**:
  ```bash
  # Decide which is canonical, then:
  git mv db/schema-v2.ts db/schema.v2.draft.ts   # or delete if abandoned
  # Add a one-line header comment to db/schema.ts:
  #   "Canonical schema — source of truth for `npx drizzle-kit push`."
  ```
- **Priority**: P1 · **Effort**: S

### 3.2 Code Quality & TypeScript Usage

- **Issue**: `db/index.ts` types the Drizzle instance as `let db: any = null`.
- **Impact**: Every `db.select().from(siddhis)` call site loses column-level type checking and
  autocomplete — the exact safety Drizzle exists to provide.
- **Solution**:
  ```ts
  // db/index.ts
  import { drizzle, type NodePgDatabase } from "drizzle-orm/node-postgres";
  import { Pool } from "pg";
  import * as schema from "./schema";

  let db: NodePgDatabase<typeof schema> | null = null;
  // ...
  if (databaseUrl) {
    // ...
    db = drizzle(pool, { schema });
  }
  export { pool, db };
  ```
- **Priority**: P1 · **Effort**: S

- **Issue**: `tsconfig.json` has `"strict": false`.
- **Impact**: Null/undefined-unsafety and implicit `any` bugs (like the one above) go undetected by
  the type checker even though the codebase is clean enough today to likely pass strict mode with
  modest fixup.
- **Solution**: Flip incrementally rather than all at once:
  ```json
  { "compilerOptions": { "strict": true, "noUncheckedIndexedAccess": true } }
  ```
  then run `npx tsc --noEmit` and fix file-by-file (budget ~1 day given current codebase size).
- **Priority**: P2 · **Effort**: M

### 3.3 Performance (Frontend + Backend)

- **Issue**: Hero images used raw `<img>` instead of `next/image`.
- **Impact**: No automatic `srcset`/format negotiation (AVIF/WebP), no `sizes`-aware download, worse
  LCP on mobile — directly hurts Core Web Vitals, which factor into Google ranking.
- **Solution**: ✅ **Applied** in `app/page.tsx` and `app/oracle/components/Hero.tsx` (switched to
  `<Image fill priority sizes="100vw" />`), plus registered the remote host in
  `next.config.ts` → `images.remotePatterns`.
- **Priority**: P0 · **Effort**: S

- **Issue**: Every API route and `app/sitemap.ts` declares `export const dynamic = "force-dynamic"`,
  and `ensureArchiveSeeded()` runs on every request via `layout.tsx`.
- **Impact**: Content that changes rarely (siddhi folios, manuscripts, the sitemap) is refetched and
  reseeded on every single request, adding avoidable DB round-trips to every page load.
- **Solution**: Move to ISR for read-heavy, rarely-changing routes:
  ```ts
  // app/sitemap.ts
  export const revalidate = 3600; // 1 hour, instead of force-dynamic

  // app/api/archivist/route.ts — cache the corpus, not the search
  import { unstable_cache } from "next/cache";
  const getCorpus = unstable_cache(
    async () => ({
      siddhis: await db.select().from(siddhis),
      manuscripts: await db.select().from(manuscripts),
    }),
    ["archivist-corpus"],
    { revalidate: 300 }
  );
  ```
  and hoist `ensureArchiveSeeded()` out of the per-request path into a one-time boot check (e.g. a
  module-level promise memoized across warm lambda/server invocations).
- **Priority**: P1 · **Effort**: M

- **Issue**: `/api/archivist` loads *all* siddhi + manuscript rows and does an O(n·terms) substring
  scan in JS on every request, duplicating logic that `/api/knowledge`'s SQLite FTS5 index already
  solves properly.
- **Impact**: Won't scale past a few hundred rows; also two different, inconsistent search
  implementations to maintain.
- **Solution**: Either (a) port `/api/archivist` to query the same FTS5 `knowledge.db` used by
  `/api/knowledge`, or (b) add a Postgres `tsvector` GIN index on `siddhis`/`manuscripts` and use
  `to_tsquery` via Drizzle's `sql` helper instead of JS-side scoring.
- **Priority**: P2 · **Effort**: M

### 3.4 SEO (Technical + Content)

- **Issue**: No `robots.txt`, and `sitemap.ts` hardcoded `astrokalki.example.com`.
- **Impact**: Search engines had no explicit crawl directive, and the sitemap would have shipped a
  fake domain to Google Search Console in any real deployment.
- **Solution**: ✅ **Applied** — added `app/robots.ts` (allows `/`, disallows `/api/`, points to
  `/sitemap.xml`), and `app/sitemap.ts` now reads `NEXT_PUBLIC_SITE_URL` from the environment (see
  updated `.env.example`).
- **Priority**: P0 · **Effort**: S

- **Issue**: `metadata` in `app/layout.tsx` only sets `title`/`description` — no Open Graph, Twitter
  card, or canonical URL, so shares to WhatsApp/X/YouTube-description-links render with no preview
  image or wrong title.
- **Impact**: Directly hurts click-through from the social/YouTube funnel the user explicitly cares
  about (their stated business is YouTube SEO & automation tools).
- **Solution**:
  ```ts
  // app/layout.tsx
  export const metadata: Metadata = {
    metadataBase: new URL(process.env.NEXT_PUBLIC_SITE_URL ?? "https://astrokalki.example.com"),
    title: { default: "AstroKalki · The Living Archive", template: "%s · AstroKalki" },
    description: "...",
    openGraph: {
      type: "website",
      images: [{ url: "/og-default.png", width: 1200, height: 630 }],
    },
    twitter: { card: "summary_large_image" },
  };
  ```
  Repeat per-route `generateMetadata()` for `app/siddhi/[slug]/page.tsx`,
  `app/manuscripts/[slug]/page.tsx`, `app/reader/[slug]/page.tsx` so each folio has a unique
  title/description/OG image built from its own data (huge win — 35 siddhis + 11 manuscripts become
  35+11 indexable, shareable pages instead of generic ones).
- **Priority**: P1 · **Effort**: M

### 3.5 AI Integration & Automation

- **Issue**: `/api/archivist/semantic` requires the *caller* to already supply a query embedding —
  no client in the codebase ever calls it, so the feature is effectively dead code.
- **Impact**: A pgvector-backed semantic search was clearly planned (schema + endpoint + seed script
  exist) but never wired end-to-end, so users only get keyword search.
- **Solution**: Add a tiny server-side embedding step so the client just sends text:
  ```ts
  // app/api/archivist/semantic/route.ts
  const queryEmbedding = await getEmbedding(query); // wraps OpenAI text-embedding-3-small
  // then run the existing pgvector <=> similarity query
  ```
  and add a "Deep search" toggle in the Archivist UI (`components/Archivist.tsx`) that calls this
  route instead of the keyword one.
- **Priority**: P2 · **Effort**: M

- **Issue**: `scripts/generate_embeddings.py` and `scripts/generate_docs.py` exist but aren't part of
  any automated pipeline (no npm script, no CI job, no seed hook).
- **Impact**: Docs and embeddings silently go stale as content is added to `knowledge/`.
- **Solution**: Add npm script wrappers and a manual-dispatch GitHub Action:
  ```json
  // package.json
  "scripts": {
    "knowledge:embed": "python3 scripts/generate_embeddings.py",
    "docs:generate": "python3 scripts/generate_docs.py"
  }
  ```
- **Priority**: P3 · **Effort**: S

### 3.9 Scalability

- **Issue**: `ensureArchiveSeeded()` runs inside `RootLayout` on every server render (`app/layout.tsx`
  is an `async function` awaiting it unconditionally, guarded only by a try/catch).
- **Impact**: Every page navigation pays for a schema-existence check + potential seed query before
  rendering the nav slugs — pure overhead once the DB is already seeded (the common case).
- **Solution**: Memoize the "already seeded" result at the module level so it only actually hits the
  DB once per server process:
  ```ts
  // lib/bootstrap.ts
  let seeded: Promise<void> | null = null;
  export function ensureArchiveSeeded() {
    if (!seeded) seeded = doEnsureArchiveSeeded();
    return seeded;
  }
  ```
- **Priority**: P1 · **Effort**: S

- **Issue**: No pagination anywhere — `/archive`, `/api/archivist`, and the sitemap all fetch entire
  tables.
- **Impact**: Fine at 35 siddhis / 11 manuscripts today; will not scale gracefully if the corpus
  grows 5-10x (a stated goal, given the extensive `knowledge/tantra/*.pdf` source library still being
  mined for content).
- **Solution**: Add `limit`/`cursor` params to `/api/archivist` and switch `/archive` to
  `generateStaticParams` + ISR once corpus size passes ~100 entries.
- **Priority**: P3 · **Effort**: M

## 4. 10x Vision

### 4.1 Turn the Knowledge Corpus into a Real RAG-Powered Archivist

- **Description**: The repo already has everything needed for retrieval-augmented generation — a
  chunked JSONL corpus (`knowledge/chunks/*.jsonl`), a metadata index, a SQLite FTS5 database, a
  pgvector-ready schema, and an embedding generation script. Wire these into a genuine RAG pipeline:
  embed on ingest → hybrid (FTS + vector) retrieval → LLM synthesis *with citations back to source
  manuscripts*, keeping the site's "scholarly restraint, pointers not prescriptions" ethos by having
  the LLM only ever quote/cite, never invent practice instructions.
- **Strategic Advantage**: Differentiates AstroKalki from every other spiritual-content site, which
  either has no search or a generic chatbot with no source grounding. A citation-backed AI archivist
  over a hand-curated Sanskrit/tantra corpus is a genuinely defensible moat.
- **Implementation Path**: (1) Finish wiring `/api/archivist/semantic` end-to-end (Section 3.5).
  (2) Add an LLM synthesis step behind the retrieval — a small, tightly-prompted call (Claude/GPT)
  that must only summarize the top-k retrieved chunks with inline `[source: slug]` citations, never
  free-generate. (3) Cache synthesized answers per-query in Postgres to avoid repeat LLM cost.

### 4.2 Per-Folio SEO Pages with Auto-Generated Rich Snippets — ✅ Siddhi/manuscript metadata + OG images shipped

- **Description**: Each of the 35 siddhi folios and 11 manuscripts already has structured data
  (`SiddhiJsonLd.tsx` exists) but the pages share generic top-level metadata. Auto-generate unique
  `generateMetadata()`, OG images (via `@vercel/og` / `next/og`), and FAQ schema per folio from
  existing DB fields (`faq`, `summary`, `primaryMantra`).
- **Strategic Advantage**: 46+ uniquely-optimized, shareable, rich-snippet-eligible pages instead of
  one generic homepage for Google — this is the single highest-leverage SEO move available given the
  content already exists in the database.
- **Implementation Path**: (1) Add `app/siddhi/[slug]/opengraph-image.tsx` using `next/og` to render
  a branded card per siddhi (name, Sanskrit, category badge). (2) Extend `generateMetadata` per
  dynamic route. (3) Submit the now-dynamic sitemap to Google Search Console.

### 4.3 "Living Archive" Progressive Web App with True Offline Practice Mode

- **Description**: The PWA scaffolding (manifest, service worker, offline page) already exists but
  was silently broken (Section 3.6) and only precaches navigation shells, not the actual practice
  data. Extend it to precache the full siddhi/manuscript JSON payload and enable the japa
  counter/timer/breath tools to work fully offline — genuinely valuable for meditation retreats or
  low-connectivity users, and a strong "install this app" pitch.
- **Strategic Advantage**: No competing spiritual-practice site offers a fully offline, no-account,
  encrypted-at-rest practice companion — this is a unique, privacy-forward positioning.
- **Implementation Path**: (1) With the sw.js fix already shipped, add an IndexedDB-backed cache of
  the full siddhi/manuscript corpus on first load. (2) Add a "Download for offline" button per
  section. (3) Verify Lighthouse PWA score reaches 100.

### 4.4 YouTube-to-Archive Content Loop

- **Description**: Given the user's stated YouTube SEO/automation business, build a lightweight
  bridge: each siddhi folio gets a canonical, timestamp-friendly URL and an auto-generated
  "video script outline" (title, hook, key citations) exportable as Markdown — turning the archive
  into a content-production accelerator for a companion YouTube channel, and the YouTube video
  descriptions link back to the folio for SEO backlink value.
- **Strategic Advantage**: Converts a content archive that currently only serves direct visitors into
  a two-way growth engine with an existing, stated business line.
- **Implementation Path**: (1) Add an "Export as video outline" action on `/siddhi/[slug]` that
  assembles existing fields (summary, benefits, warnings, primary mantra) into a script skeleton.
  (2) Add the RSS feed from Section 3.10 so new folios can be auto-queued into a video pipeline.

### 4.5 Comparative Tradition Graph as an Interactive Knowledge Graph

- **Description**: The `/comparative` 5-tradition matrix and `/lineage` tree are currently separate,
  static views. Combine them into one force-directed, explorable knowledge graph (siddhi ↔
  tradition ↔ lineage ↔ manuscript ↔ evidence source) using the existing
  `ForceDirectedConstellation.tsx` component as a base, with the semantic search from 4.1 able to
  highlight/filter the graph live.
- **Strategic Advantage**: A genuinely novel visualization for this content category — most
  comparative-religion resources are just tables or prose. This becomes the site's signature
